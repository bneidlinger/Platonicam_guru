Metadata Strategy: "The King of Context"

For an Enterprise Physical Security Engineer, accuracy is non-negotiable. This document defines how we use Structured Metadata to augment Unstructured AI Chat.

The Core Principle: Use LLMs for summarization and natural language; use Metadata for computation and verification.

1. Metadata Schema Definition

Tier 1: Document
  • vendor: Hanwha, Axis, Bosch, etc.
  • doc_type: Datasheet, Installation, Accessory.
  • source_url: Path to local PDF file.
Tier 2: Engineering
  • model_num: Primary search key (Regex-extracted).
  • poe_wattage: Numerical value for power budgets.
  • poe_class: IEEE standard classification.
Tier 3: Visual
  • image_refs: Paths to extracted PNG assets.
  • page_num: Page index for source verification.
  • bbox: (Optional) Coordinates of tables/images.

2. Data Structure Example (JSON)

This is how a single record appears in our Vector Database (ChromaDB):

{
    "content": "The Axis P3265-LVE features a max power consumption of 12.9W (PoE Class 3). It is an outdoor-ready dome with Forensic WDR.",
    "metadata": {
        "model_num": "P3265-LVE",
        "brand": "Axis",
        "poe_wattage": 12.9,
        "poe_class": "3",
        "form_factor": "Dome",
        "images": ["./assets/images/P3265_hero.jpg"],
        "page_source": 4
    }
}

3. Practical Logic Workflows

A. Automated PoE Budgeting

The system iterates through the project_list, retrieves the poe_wattage metadata for each model, and returns a hard mathematical sum. No LLM "hallucination" of numbers is possible.

B. Accessory Verification

When searching for mounts, the system filters by model_num + doc_type: "Accessory Guide". It then renders the associated image_refs so the engineer can visually confirm the bracket shape.
